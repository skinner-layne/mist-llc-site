Product

What Mist does.

Mist sits between your application and every model provider. It accepts an OpenAI-compatible request, picks the right model based on task type and cost, and returns a response. If a provider fails, it transparently retries on another. Every decision is logged.

Architecture

How a request flows through Mist

Five steps, 60 milliseconds of routing overhead, no SDK changes on your side.

  1. Your application sends a request

    Your code uses the OpenAI client library, pointed at api.mist.llc/v1/chat/completions. The request body is identical to what you’d send to OpenAI directly.

  2. Mist classifies the task

    The router inspects the prompt, system message, and any explicit task hint header. It assigns one of 47 task types: reasoning, code generation, summarization, vision, OCR, classification, etc.

  3. Mist picks a model

    For the assigned task type, Mist looks up the active routing chain — ordered by cost-quality ratio, filtered by your project’s allowed providers and any data-residency constraints. The top-ranked model wins.

  4. Mist calls the provider

    Mist forwards the request to the chosen provider over a commercially-papered enterprise account. If the provider returns 5xx, hits a rate limit, or exceeds latency budget, Mist retries on the next model in the chain.

  5. Mist returns the response and logs everything

    Your application receives an OpenAI-format response. In parallel, Mist logs the prompt hash, the routing decision, the provider, the latency, the token counts, and the cost — available in your dashboard and exportable to your SIEM.

Routing logic

Task types, cost optimization, fallback chains

Task-type classification

Mist supports 47 task types. Each represents a distinct shape of AI work, from high-stakes reasoning down to bulk classification. Common ones: reasoning, code, creative, synthesis, summarization, quick, vision, ocr, web_search, math_science.

You can let Mist classify automatically (default) or pin a task type explicitly via the X-Mist-Task header.

Cost optimization

Within each task type, models are ordered by a cost-quality score derived from continuous benchmarking. Mist picks the cheapest model that meets your minimum quality threshold for that task.

You can override this per-project: pin to a specific provider, set a maximum cost-per-million-tokens, or require a specific compliance tier.

Fallback chains

Every task type has an ordered fallback chain. If the primary fails (5xx, rate limit, latency budget exceeded), Mist tries the next model in the chain. Customers see one successful response; the failover is invisible.

Routing overrides

Quality regressions get caught fast. Our Assayer system continuously samples production traffic, scores responses, and updates routing weights nightly. When a model degrades, traffic shifts away within hours.

Drop-in API

OpenAI-compatible. Three lines of change.

If your code calls OpenAI today, the migration is a base URL and an API key. No SDK swap. No request shape changes. No response shape changes.

Python

from openai import OpenAI

client = OpenAI(
    api_key="sk-mist-...",
    base_url="https://api.mist.llc/v1",
)

response = client.chat.completions.create(
    model="auto",  # let Mist pick
    messages=[
        {"role": "user", "content": "Explain merge sort in 3 sentences."}
    ],
)

print(response.choices[0].message.content)

Node.js

import OpenAI from "openai";

const client = new OpenAI({
  apiKey: process.env.MIST_API_KEY,
  baseURL: "https://api.mist.llc/v1",
});

const response = await client.chat.completions.create({
  model: "auto",
  messages: [{ role: "user", content: "Summarize the Treaty of Westphalia." }],
});

console.log(response.choices[0].message.content);

curl

curl https://api.mist.llc/v1/chat/completions \
  -H "Authorization: Bearer $MIST_API_KEY" \
  -H "Content-Type: application/json" \
  -H "X-Mist-Task: reasoning" \
  -d '{
    "model": "auto",
    "messages": [{"role": "user", "content": "What is the time complexity of quicksort?"}]
  }'
