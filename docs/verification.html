CONCEPTS

Prism verification

Prism runs the same prompt through multiple uncorrelated models in parallel, compares the answers, and returns a verified result with disagreement surfaced. Use it when being wrong costs more than being slow.

Why verification exists

Any single LLM has a non-zero rate of confident wrong answers. In many production systems that is acceptable: the model is right often enough, the output is advisory rather than decisive, and the cost of an occasional bad answer is low. A routed single-model call is usually the correct tradeoff for speed, cost, and throughput. For use cases like autocomplete, internal search assistance, or low-risk classification, adding more checks often creates more overhead than value.

Some workloads have a different error budget. Financial decisions, medical triage, legal review, and any irreversible action can make a single incorrect answer materially expensive. For those requests, a single-model response is often not safe enough on its own. Prism is Mist's answer to that problem. Instead of calling one model, Prism runs the prompt through 3–8 uncorrelated models in parallel and compares the answers. If they agree, you get a high-confidence result. If they disagree, you get the disagreement surfaced for review. This is not free: it costs more, takes longer, and is only worth using where the cost of being wrong dominates.

How it works

  1. Mist receives the request with verification: "prism" set.
  2. Mist selects 3–8 models from the routing table, ensuring no two share a base model lineage.
  3. The same prompt is dispatched in parallel to each selected model.
  4. Each response is collected and normalized.
  5. A judge model compares the responses and identifies the consensus answer.
  6. The judge also identifies any meaningful disagreements.
  7. Mist returns a single response with the consensus answer plus a prism_metadata object containing all individual responses, the agreement score, and any disagreement notes.

When to use Prism

  • The cost of a wrong answer exceeds the cost of additional latency and tokens.
  • You're making an irreversible decision based on the model output.
  • You're in a regulated context where you need an audit trail showing multiple models agreed.
  • You're spot-checking a new model before promoting it to production.
  • You're investigating a suspicious answer from a single-model call.

Do not use Prism for high-volume, low-stakes requests. The latency and cost overhead make it impractical for workloads like autocomplete, chat triage, or bulk classification. For those, single-model routing is the right choice.

Cost and latency

A Prism call invokes 3–8 models plus a judge. In practice, cost is roughly 4–9× a single-model call. Latency is bounded by the slowest model in the panel, so end-to-end time is typically 2–4× a single-model call because the panel runs in parallel. Pricing is still token-based across all panel members, and you can inspect the final breakdown in prism_metadata.cost_usd.

Prism is intentionally expensive. Use it where the math justifies it. For everything else, use single-model routing and check the audit log if a specific request looks wrong.

Configuring Prism

The simplest way to use Prism is to set verification: "prism" in the request body. Mist picks the panel for you using the built-in diversity rules.

curl https://api.mist.llc/v1/chat/completions \
  -H "Authorization: Bearer $MIST_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "auto",
    "verification": "prism",
    "messages": [
      {
        "role": "system",
        "content": "Answer concisely and cite uncertainty."
      },
      {
        "role": "user",
        "content": "Based on the attached intake notes, should this claim be escalated for manual review?"
      }
    ]
  }'

If you want to control the panel yourself, pass verification: "prism" along with a verification_panel array of model names. Mist will dispatch only to those models. This is useful for benchmarking and for compliance scenarios where the panel composition is fixed by policy.

Reading the result

The top-level response still matches the OpenAI envelope, so existing code keeps working. In particular, response.choices[0].message.content still gives you the consensus answer. Mist adds a prism_metadata field with the verification details.

Use agreement_score as an operational signal rather than a guarantee. Scores below 0.7 should usually escalate to human review. Scores from 0.7 to 0.95 are best treated as verified with caveats, which means you should read the disagreements field. Scores above 0.95 are typically safe to treat as verified consensus.

Other verification tiers

Mist supports several verification tiers; Prism is the production default. Other tiers are documented in the API reference once they ship:

  • none — single-model routing, no verification (default)
  • single — single-model with a re-read check by a judge
  • prism — 3–8 model parallel panel (this page)
  • debate — affirmative-and-negative debate with a judge (high latency, high quality)
  • parliament — 8–50 model deliberation (used for the most critical decisions)
  • cascade — adaptive multi-round verification (used when an early disagreement signals deeper investigation is needed)

Each tier trades latency and cost for verification depth. Most teams start with prism for critical traffic and none for everything else.
