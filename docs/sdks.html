SDKs and client libraries

Mist offers official client libraries for the languages enterprise teams use most. Or use the OpenAI client library you already have — Mist is fully OpenAI-compatible.

Choose your path

You have three supported integration options: use the official Mist SDK for your language, use the OpenAI client library with Mist's base URL set, or call the REST API directly with curl or your HTTP library of choice. For new projects, use the Mist SDK. For migrations, using the OpenAI client is often the fastest path. All three options are supported. The Mist SDK adds typed access to Mist-specific fields such as verification tiers, task types, prism_metadata, and fallback chains, while the OpenAI client works for everything that is OpenAI-compatible.

Python

The Python SDK is the most complete Mist client today and is the recommended choice for production Python services.

Install: pip install mist

from mist import Mist

client = Mist(api_key="mist_live_pk_...")

response = client.chat.completions.create(
    model="auto",
    task_type="reasoning",
    messages=[
        {"role": "user", "content": "Explain quicksort in two sentences."}
    ],
)

print(response.choices[0].message.content)
print(response.mist_metadata.routed_to)

The client is thread-safe. Async support is available via from mist import AsyncMist. If no key is passed explicitly, the SDK reads MIST_API_KEY automatically.

Node.js / TypeScript

The Node.js SDK exposes the same API shape as the HTTP service with typed request and response objects for Mist-specific fields.

Install: npm install @mist/sdk

import Mist from "@mist/sdk";

const client = new Mist({ apiKey: process.env.MIST_API_KEY });

const response = await client.chat.completions.create({
  model: "auto",
  task_type: "code",
  messages: [
    { role: "user", content: "Write a TypeScript function that debounces a callback." },
  ],
});

console.log(response.choices[0].message.content);
console.log(response.mist_metadata.routed_to);

The SDK is fully typed. ESM and CommonJS are both supported. It works in Node.js 18+ and modern bundlers. Browser usage is not supported because API keys must remain server-side.

Go

The Go SDK is designed around standard Go patterns and is a good fit for backend services that need explicit context propagation and straightforward error handling.

Install: go get mist.llc/mist-go

package main

import (
    "context"
    "fmt"
    "log"

    "mist.llc/mist-go"
)

func main() {
    client := mist.NewClient("mist_live_pk_...")

    resp, err := client.ChatCompletions.Create(context.Background(), &mist.ChatCompletionRequest{
        Model:    "auto",
        TaskType: "summarization",
        Messages: []mist.Message{
            {Role: "user", Content: "Summarize this changelog: ..."},
        },
    })
    if err != nil {
        log.Fatal(err)
    }

    fmt.Println(resp.Choices[0].Message.Content)
}

The SDK is context-aware, returns errors directly, and does not use panics in the happy path. It works with Go 1.21+.

Using the OpenAI client library

If you already use the official OpenAI SDKs, migration can be as simple as changing the base URL and API key. Mist speaks the OpenAI API, so existing call sites often need little or no structural change.

from openai import OpenAI
client = OpenAI(
    base_url="https://api.mist.llc/v1",
    api_key="mist_live_pk_...",
)
import OpenAI from "openai";
const client = new OpenAI({
  baseURL: "https://api.mist.llc/v1",
  apiKey: process.env.MIST_API_KEY,
});

The trade-off is that the OpenAI client gives you a low-friction migration path, but it does not expose Mist-specific fields such as verification or task_type as native typed properties. You can still pass them through extra_body in Python or by extending the request object in TypeScript, but typed access requires a Mist SDK.

Versioning

SDK versions follow semantic versioning. Major version bumps are reserved for incompatible API changes, and those changes also require a corresponding API version bump on the server side. Minor version bumps add features without breaking existing code. Patch version bumps are limited to bug fixes and internal improvements.

Each repository includes its own compatibility matrix in the README, including the supported SDK versions, API versions, and minimum runtime requirements.
