CONCEPTS

How routing works

Mist routes each request to the cheapest model that can handle it, with automatic failover when a provider degrades. This page explains how that decision gets made.

The mental model

Mist sits between your application and more than 20 model providers. Your application sends one request to the Mist API, and Mist decides where to send it upstream. The primary input to that decision is the task type: a hint you include in the request body to describe the kind of work being done. Mist looks up that task type in a routing table, which maps it to an ordered chain of provider and model candidates, then dispatches to the first entry in the chain.

If the selected provider returns an error, is rate-limited, or times out, Mist retries against the next entry in the chain without requiring any change in your client code. Your application receives either one successful response or one terminal error after the chain is exhausted. This is not magic: the routing table is configurable, fallback behavior is documented, and every routing decision is logged in the audit trail so you can query it later by request_id.

Task types

Mist supports 47 task types organized into seven tiers. The task type is a hint your application passes in the request body. If you do not provide one, Mist uses a sensible default: default, which maps to a fast general-purpose model. If you do provide a hint, Mist selects from a curated chain of models chosen for that task class.

Task type (code) Description Typical first-choice model
reasoning Complex chain-of-thought, math, planning frontier reasoning model
code Code generation and refactoring specialized code model
creative Long-form writing, marketing copy large general model
quick_response Triage, classification, short answers fast inference model
summarization Document and conversation summarization fast general model
vision Multimodal image understanding vision-capable model
web_search Queries that require live web data search-augmented model
default General-purpose, no special hint balanced general model

The exact provider and model mapping is defined in the live routing table and may change as upstream vendors ship new models, change pricing, or degrade in production.

Fallback chains

Every task type maps to an ordered chain of model candidates. That chain is the failover path. If the first model errors, times out, or returns an unparseable response, Mist automatically tries the next model in the sequence, then the next, until one succeeds or the chain is exhausted.

  1. HTTP 5xx from the upstream provider
  2. HTTP 429 (rate limit) from the upstream provider
  3. Connection timeout (configurable, default 60s)
  4. Empty or malformed response body
  5. Provider explicit "service degraded" signal

By contrast, 4xx errors caused by your request are not retried. If you send invalid input, an unsupported parameter, or a malformed payload, Mist returns that error directly to your application. Only failures attributable to the provider trigger fallback.

Picking a specific model

If you do not want routing, pass a concrete model name in the model field. Mist will dispatch directly to that model and only that model. Fallback is disabled in this mode unless you also pass an explicit fallback_chain array. Use direct model selection when you have a hard dependency on a model's behavior or when you are benchmarking across providers.

Cost optimization

The default routing tier balances cost and quality. If you want strict cost minimization, use task_type: "quick_response" for work that does not require deeper reasoning. If you want strict quality maximization, use task_type: "reasoning" even for simpler prompts, understanding that this usually increases cost and latency.

Most teams see 30–60% cost reduction in the first month just by switching their non-critical traffic to quick_response. Reserve reasoning for the queries where being right matters more than being cheap.

Rate limits and quotas

Each API key has a per-minute and per-day request quota negotiated at sales. When you exceed the quota, Mist returns 429 with a Retry-After header in milliseconds. Rate limits are per key, not per endpoint. Enterprise contracts include burst headroom; talk to sales if you need a custom limit.

Audit trail

Every routing decision is logged with the request ID, the task type you sent, the chain Mist evaluated, the model that ultimately served, the latency, and the cost. The log is retained per your contract and exportable to your SIEM.

Common patterns

  • Pin critical paths to a specific model with model: "..." and a custom fallback_chain
  • Tag every request with a task_type hint for routing precision
  • Use Idempotency-Key headers for safe retries
  • Set conservative max_tokens to bound cost
  • Subscribe to the audit log webhook for real-time observability
